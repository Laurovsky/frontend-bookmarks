# введение в TypeScript

**TypeScript** — строго типизированный язык программирования. Часто его называют языковой надстройкой JavaScript. Всё потому, что написанный на TypeScript код выполняется не напрямую в браузере. Сначала он компилируется в JavaScript.

## Добавляем TypeScript в проект

**Шаг 1: проверка установки Node.js и npm**

TypeScript использует npm (Node Package Manager). Поэтому первым делом нужно установить Node.js, вот [инструкция](https://nodejs.org/en) по установке.

После установки нужно убедиться, что всё в порядке. Для этого запустим эти программы в терминале:
```
node -v
npm -v
```

**Шаг 2: создание папки проекта**

Создайте новую папку и перейдите в неё. Сделать это можно любым удобным для вас способом: через проводник или терминал.

Если вы выбрали терминал, то в нём нужно выполнить следующие команды:
```
mkdir my-ts-project
cd my-ts-project
```

**Шаг 3: инициализация проекта и установка TypeScript**

Вы создадите файл `package.json` с базовой конфигурацией проекта. Это нужно для того, чтобы вы могли устанавливать в проект зависимости, ведь TypeScript — это такой же npm-пакет.

Для установки запустим команду:
```
npm install --save-dev typescript
```

**Шаг 4: настройка TypeScript (`tsconfig.json`)**

После установки нужно настроить TypeScript — указать, какой код проверять и насколько, а какой стоит игнорировать. Это лишь малая часть возможностей, которые есть у настройки TypeScript.

Для создания базовой конфигурации, или, другими словами, — конфига, запустим команду:
```
npx tsc --init
```

Созданный файл `tsconfig.json` — и есть конфиг проекта:

**Шаг 5: создание первого TypeScript-скрипта**

У файла должно быть расширение `.ts` — так компилятор поймёт, что в файле находится TypeScript-код. Это основное расширение для TypeScript файлов, но есть и другие. С ними вы будете знакомиться по мере изучения программы.

Создадим директорию `src`, а в ней файл `index.ts`. Сделать это можно любым удобным способом, например, с помощью команд в терминале:
```
mkdir src
touch index.ts
```

Теперь в этот файл добавим содержимое:
```
function greet(name: string): string {
  return `Привет, ${name}!`;
}

console.log(greet("Алекс"));
```

Обратите внимание: мы указали, что аргумент `name` должен быть строкой (`string`). Это и есть статическая типизация.

**Шаг 6: компилируем код в JavaScript**

На этом шаге наша задача — превратить TypeScript-код в аналог на JavaScript. Сделать это нужно потому, что браузер не умеет работать с TypeScript напрямую.

Запустим эту команду:
```
npx tsc
```

Команда создаст файл `index.js` рядом с `index.ts`.

Содержимое `index.js` будет таким:
```
"use strict";
function greet(name) {
    return `Привет, ${name}!`;
}
console.log(greet("Алекс"));
```
TypeScript удалил типы, так как они нужны только при разработке и не могут использоваться в JavaScript-коде.

## Команда в package.json

Обычно команды сборки и запуска намного сложнее тех, которые мы выполняли в рамках этого урока. Запускать их также приходится довольно часто. Поэтому их можно записывать в поле `scripts` в `package.json` — так вам не придётся каждый раз создавать их заново:
```
// package.json
{
  ...
  "scripts": {
    "build": "tsc"
  },
  ...
}
```
Теперь, если открыть консоль и выполнить команду `npm run build`, запустится скрипт `build` из `package.json`.

## Настройка TypeScript

### Целевая директория

Файлы с исходным кодом и скомпилированные файлы должны храниться отдельно.

- исходный код нужно положить в одну директорию;
- скомпилированный — в другую.

Так делают, чтобы скомпилированный код можно было легко «съесть» — перенести на публичный сервер.

Директорию с исходным кодом обычно называют `src`. Код, который получается в результате компиляции и сборки, складывают в директорию `dist`.

Проблема в том, что TypeScript по умолчанию кладёт их рядом. При таком подходе неудобно добавлять собранные файлы в список исключений системы контроля версий и выгружать их на сервер.

Намного лучше положить все артефакты компиляции отдельно. Для этого укажите параметр `outDir` в конфиге, а точнее — в его разделе `compilerOptions`:
```
{
  "compilerOptions": {
    "outDir": "dist"
  }
}
```

### Включение и исключение файлов
Если запустить компилятор `tsc`, не указывая конкретный файл в качестве аргумента, то в сборку попадут все TypeScript-файлы проекта — это вы уже знаете.

Но что, если не все файлы нужно собирать? В TypeScript есть целых три параметра, которые помогут объяснить компилятору, какие файлы обрабатывать. Все они указываются в корне конфига (не в блоке `compilerOptions` ).

С помощью параметра `files` можно указать конкретный список файлов для компиляции:
```
{
  "compilerOptions": {},
  "files": [
    ".src/core.ts",
    ".src/app.ts"
  ]
}
```
Компилятор соберёт именно эти файлы, а также все, которые в них подключены.

Проблема в том, что большие проекты состоят из множества файлов, и перечислять их вручную неудобно.

Упростит жизнь другой параметр — `include`. В нём указывают шаблоны файлов и путей к файлам, которые должны быть скомпилированы.

Вот как это работает: одна звёздочка ( `*` ) обозначает любую последовательность символов. Две ( `**` ) — любую папку или подпапку. Так шаблоном можно заменить все возможные пути. Например, с помощью такого кода можно сказать компилятору: «Собери только те файлы, которые лежат в папках и подпапках src и tests »:
```
{
  "compilerOptions": {},
  "include": [
    "src/**/*",
    "tests/**/*"
  ]
}
```

Это займёт немного больше времени, чем использование `files`, — искать файлы по шаблону дольше, чем обрабатывать готовый список. Но это увеличение по времени не так существенно, как удобство, которое несёт в себе `include`.

Если в файлы для компиляции попало что-то лишнее, поможет параметр `exclude` — ему тоже можно передать шаблоны файлов:
```
{
  "compilerOptions": {},
  "exclude": [
    "node_modules",
    "./src/**/*.spec.ts"
  ]
}
```

По умолчанию этот параметр включает папки модулей ( `node_modules` ) и директорию компиляции (указанную в параметре `outDir` ). Если будете переопределять значение `exclude`, не забудьте вручную исключить эти пути.

### Смешанный код

Перевод кодовой базы проекта с JavaScript на TypeScript — процесс небыстрый. И чтобы в это время код оставался рабочим, нужно научить компилятор принимать на вход как JS-, так и TS-файлы. Сделать это поможет параметр `allowJs`:
```
{
  "compilerOptions": {
    "allowJs": true
  }
}
```

Теперь не нужно полностью переводить проект на TS, чтобы продолжать работу. Можно делать это постепенно, файл за файлом, а новый код писать сразу на TypeScript.

### Строгий режим

В TypeScript есть строгий режим, включается он параметром `strict`:
```
{
  "compilerOptions": {
    "strict": true
  }
}
```

В строгом режиме компилятор включает дополнительные правила проверки кода. Это снижает вероятность того, что в итоговом коде появятся ошибки.

Вот правила строгого режима:
- `noImplicitAny` запрещает неявный тип `any`;
- `noImplicitThis` запрещает неявно определять `this` -контекст;
- `strictNullChecks` проверяет, могут ли используемые значения быть `null` и `undefined`;
- `strictPropertyInitialization` следит, чтобы в классе все свойства были инициализированы;
- `strictBindCallApply` проверяет, чтобы `bind`, `call` и `apply` были использованы с правильными аргументами;
- `strictFunctionTypes` сравнивает функции по контравариантным правилам.

Правила можно включить по отдельности, не включая строгий режим.

### Файлы декларации и Source Maps

TypeScript содержит типы, но при компиляции они пропадают, — в JS-коде остаётся только сухая реализация. Если под рукой нет исходных файлов, то непонятно, какие типы в них были изначально.

Когда в исходных файлах проекта импортируется какая-то функциональность из установленного пакета, компилятор смотрит в папку `node_modules`. Там лежит уже скомпилированный код, а значит, вся типизация пакета исчезла.

Для обратной совместимости рядом с собранным файлом кладут файл декларации с разрешением `.d.ts` и таким же именем. В этом файле, в отличие от JS-файла, содержатся только описания типов, но нет реализации.

TypeScript умеет сам создавать такие файлы на этапе компиляции. Для этого нужно включить опцию `declaration`:
```
{
  "compilerOptions": {
    "declaration": true
  }
}
```

Файлы Source Maps с расширением . `js.map` помогают во время отладки приложения. Они связывают строки собранного кода с исходным. Такие файлы, конечно, не нужно писать вручную — компилятор сделает это сам, если включить параметр `sourceMap`:
```
{
  "compilerOptions": {
    "sourceMap": true
  }
}
```

### Опции как аргументы

Параметры компиляции зачастую записывают в конфиг, но их также можно передавать прямо при запуске команды `tsc`:
```
tsc --sourceMap
```
Это полезно, когда нужно выполнить компиляцию с небольшими изменениями в конфиге.

### Оптимизация

Данные последней компиляции TypeScript умеет сохранять в файл `tsbuildinfo`. Это сильно ускоряет последующую компиляцию — не нужно собирать всё с нуля. Для этого включите опцию `incremental`:
```
{
  "compilerOptions": {
    "incremental": true
  }
}
```

## Vite и Typescript

Vite — это современный инструмент сборки проектов. Он работает очень быстро благодаря тому, что он использует ES модули в браузере. А ещё он отлично поддерживает TypeScript «из коробки».

Для создания проекта на Vite и Typescript нужно проделать четыре шага:

**Шаг 1: Откройте терминал.** Выберите папку, в которой вы создадите проект, и откройте терминал в этой директории.

**Шаг 2: Создайте проект с помощью Vite.** В терминале выполните следующую команду:
```
npm create vite@latest my-ts-app -- --template vanilla-ts
```

В этой команде:
- `my-ts-app` — имя нового проекта;
- `--template vanilla-ts` указывает, что вы хотите использовать шаблон с TypeScript без использования дополнительных фреймворков.

Если система спросит подтверждение — нажмите `Enter` или `Y`.

**Шаг 3: Перейдите в папку проекта и установите зависимости.** Для этого нужно выполнить команду (если на предыдущем шаге вы выбрали имя, отличное от `my-ts-app`, то именно его надо указывать в команде `cd`) — это установит все необходимые для проекта библиотеки и инструменты:
```
cd my-ts-app
npm install
```

**Шаг 4: Запустите локальный сервер разработки.** Наконец, запустим проект. Vite добавляет для этого команду `dev` в блоке `scripts` внутри `package.json`. Соответственно, для запуска надо выполнить следующую команду:
```
npm run dev
```

После выполнения команды откроется окно браузера. Адрес окна будет таким: `http://localhost:5173` — это то место, где будет работать ваш проект.

### Структура проекта

После создания проекта его структура будет примерно такой:
```
my-ts-project/
├── public/
├── src/
│   ├── main.ts
│   └── style.css
├── index.html
├── tsconfig.json
├── vite.config.ts
└── package.json
```

- `src/main.ts` — точка входа в приложение;
- `public` — место для статических ресурсов, которые нужно скопировать в финальный билд «как есть», без обработки;
- `tsconfig.json` — конфигурационный файл TypeScript;
- `vite.config.ts` — конфигурация Vite.

Помимо команды `npm run dev`, в проекте доступна команда `npm run build`. С её помощью происходит сборка проекта для выгрузки на сервер.

Но не только Typescript помогает делать проекты лучше. Для проверки кода чаще всего используется линтер ESLint. Это инструмент, который проверяет не типы, а то, как мы пишем код. Например, он запрещает использовать `var` или создавать переменные, которые не используются.

### Установка ESLint

В корне проекта выполните команду:
```
npm init @eslint/config
```

Если у вас не установлен пакет @eslint/config, то программа запросит разрешение на установку:
```
Need to install the following packages:
  @eslint/create-config
Ok to proceed? (y)
```

Введите `y`.

Затем ответьте на несколько вопросов, чтобы программа сгенерировала базовый файл конфига — для навигации и ввода ответов используйте клавиши «вверх», «вниз» и «Ввод». Сначала вас попросят указать, для чего вам ESLint. Отвечайте — для всего, что он умеет, а именно: проверять синтаксис, искать проблемы и поддерживать общий стиль написания кода:
```
? How would you like to use ESLint? …
  To check syntax only
  To check syntax and find problems
❯ To check syntax, find problems, and enforce code style
```

Следующие вопросы касаются вашего проекта. Отвечайте так, как считаете нужным.

Самый важный вопрос — «Используется ли TypeScript?» Не забудьте ответить `Yes`.
```
✔ How would you like to use ESLint? · style
✔ What type of modules does your project use? · esm
✔ Which framework does your project use? · none
 
? Does your project use TypeScript? · No / Yes
```

А теперь минутка магии: ESLint сам определит, какие пакеты и плагины ему понадобятся, и предложит их установить:
```
✔ How would you like to use ESLint? · style
✔ What type of modules does your project use? · esm
✔ Which framework does your project use? · none
✔ Does your project use TypeScript? · No / Yes
✔ Where does your code run? · browser
✔ How would you like to define a style for your project? · prompt
✔ What format do you want your config file to be in? · JavaScript
✔ What style of indentation do you use? · 4
✔ What quotes do you use for strings? · single
✔ What line endings do you use? · unix
✔ Do you require semicolons? · No / Yes

The config that you've selected requires the following dependencies:

@typescript-eslint/eslint-plugin@latest eslint-config-standard@latest eslint@^8.0.1 eslint-plugin-import@^2.25.2 eslint-plugin-n@^15.0.0 eslint-plugin-promise@^6.0.0 @typescript-eslint/parser@latest
? Would you like to install them now? › No / Yes
```

Долго не думайте, соглашайтесь! В ответе на следующий вопрос укажите, какой у вас пакетный менеджер, — и дело в шляпе.

Дождитесь установки, а затем откройте заветный файл `.eslint.config.mjs` (или `.eslint.config.js`). Его содержимое должно выглядеть примерно так:
```JS
import js from "@eslint/js";
import globals from "globals";
import tseslint from "typescript-eslint";
import { defineConfig } from "eslint/config";


export default defineConfig([
  { files: ["**/*.{js,mjs,cjs,ts,mts,cts}"], plugins: { js }, extends: ["js/recommended"] },
  { files: ["**/*.{js,mjs,cjs,ts,mts,cts}"], languageOptions: { globals: globals.browser } },
  tseslint.configs.recommended,
]);
```

Обратите внимание: в конфиге уже подключён плагин tseslint. Как раз он и содержит правила написания TypeScript-кода. С полным перечнем этих правил можно ознакомиться на официальной странице плагина. Из этих правил составлен список рекомендуемых — он подключён с помощью `tseslint.configs.recommended`.

Большинство IDE интегрированы с ESLint и проверяют код в режиме реального времени.

Проверку можно запустить и вручную. Для этого добавьте в package.json такие команды:
```
{
  "scripts": {
    ...
    "lint": "eslint src",
    "fix": "eslint --fix src"
    ...
  }
}
```
Выполните команду `npm run lint`. ESLint проверит код в директории `src` на ошибки и, если они есть — выведет информацию о них:
```
/project_1/src/index.ts
  3:41  error  Missing semicolon  semi

✖ 1 problem (1 error, 0 warnings)
  1 error and 0 warnings potentially fixable with the `--fix` option.
```

Некоторые ошибки ESLint может исправить сам — например, в коде выше он поставит пропущенную точку с запятой. Для этого выполните команду `npm run fix`.

Поздравляем, теперь у вас есть помощник. С ним ваш TypeScript станет лучше!
